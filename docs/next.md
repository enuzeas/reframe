## 오디오 먹싱 (보류, 2026-07-04 — 배선은 됐지만 실사용 불가 판정)

`output.py`의 `RTSPPublisher`에 `audio_src`(avfoundation 오디오 장치 인덱스)를 추가 —
지정하면 ffmpeg에 avfoundation 오디오 입력을 두 번째 `-i`로 추가해 Opus로 인코딩,
RTSP 스트림에 h264와 함께 먹싱한다. `sources.probe_audio_devices()`로 장치 목록(인덱스+이름,
ffmpeg 자체로만 열기 때문에 `probe_devices()`의 OpenCV/ffmpeg 인덱스 불일치 문제 없음) 조회.
`reframe-server --audio-src <idx>`/`reframe.py --audio-src <idx>`로 노출.

**결론: 실 플레이어(OBS, IINA)에서 오디오를 넣으면 영상이 멈추는 문제를 못 고쳤다.** 아래
순서로 여러 실제 버그를 찾아 고쳤고 그때마다 증상이 바뀌었지만, 오디오를 완전히 빼면 그
즉시 정상 재생되고(TCP라 약간의 지연만 있음) 다시 넣으면 재현되는 걸 여러 번 확인 — 오디오와
비디오, 두 개의 독립적인 실시간 입력을 **하나의 ffmpeg 프로세스에서 합치는 것 자체**가
근본 원인일 가능성이 높다(VLC는 화면이 나오되 버벅임 — 완전히 죽지는 않아 정도의 차이만
있었음). 시도했지만 완전히 해결 못한 것:

- [x] `-use_wallclock_as_timestamps`를 비디오에만/양쪽 다 걸어봄 — 양쪽 다 거는 쪽이
      "화면이 첫 프레임에서 완전히 멈추는" 증상은 없앴음(클럭 도메인 불일치로 판단, `start:
      -3.6...` 같은 음수 오프셋을 리더가 계산하던 게 원인)
- [x] `RTSPPublisher.close()` 행 버그 — 오디오 입력은 EOF가 없어 `proc.wait()`가 항상
      타임아웃, 처리 안 된 예외가 파이프라인 스레드를 죽임(채널 삭제만 해도 전체 서버 정지).
      `terminate()`→`kill()` 폴백으로 해결(오디오와 무관하게 유효한 수정, 유지)
- [x] `ChannelOutputs.write()` 크래시 — broken pipe 시 파이프라인 스레드가 죽던 것.
      `OSError`를 잡아 죽은 publisher만 버리고 재생성하도록 수정(유지)
- [x] `ChannelOutputs.sync()`가 채널 삭제 시 ffmpeg 종료를 동기적으로 기다려 **살아있는
      채널까지 최대 10초씩 멈추던 버그** — 백그라운드 스레드에서 닫도록 수정(유지, 오디오와
      무관하게 원래도 있던 버그. 디버그 로그로 `dt=10.571s` 직접 실측)
- [x] 트래커가 문서(`PLAN.md` §3.2)의 ByteTrack이 아니라 GMC 포함된 다른 기본값
      (`tracktrack.yaml`)으로 조용히 실행되고 있던 것 — `tracker="bytetrack.yaml"` 명시로
      GMC발 주기적 지연(최대 1초) 제거(유지, 오디오와 무관하게 유효한 수정)
- [x] mediamtx RTP 패킷 크기(1472) > 임계값(1440) — OBS가 검은 화면만 보여주던 원인.
      `-pkt_size 1200`으로 해결(유지)
- [x] RTSP를 UDP에서 TCP로 전환(퍼블리시+`mediamtx.yml`의 `rtspTransports: [tcp]`) —
      도움은 됐지만 오디오 문제 자체는 해결 못함
- [x] `-thread_queue_size 1024` 추가, 오디오를 4채널 중 1개(최소 id)에만 배정 — 그래도
      해당 채널에 오디오를 넣으면 재현됨
- **결정(사용자, 2026-07-04)**: 오디오 없이 기본값으로 되돌림. `--audio-src`는 코드에
  남아있고 필요하면 다시 시도할 수 있지만, 지금은 4채널 다 비디오 전용으로 운영.

### 실운영 해법: 오디오는 OBS에서 별도 트랙으로 (2026-07-04 확립)

먹싱을 안 하는 게 이 구조에선 오히려 **정답**이다. reframe는 같은 소스를 **4개 독립 비디오
채널**로 내보내고 OBS에서 그중 하나를 골라(전환하며) 송출하는데, 오디오는 채널과 무관하게
**하나의 연속 트랙**이어야 한다. 먹싱하면 (1) 같은 오디오가 4벌 복제되고 (2) 채널 전환 시
오디오가 끊긴다. 그래서 "비디오 4채널 = reframe / 오디오 1트랙 = OBS 직접"이 올바른 분리.

- **오디오 소스**: OBS `오디오 입력 캡처`로 마이크를 직접 추가(reframe 안 거침)
- **싱크**: reframe 파이프라인 지연으로 비디오가 오디오보다 **늦게** 도착 → OBS `고급 오디오
  속성 → 동기화 오프셋(ms)`에 **양수**를 넣어 오디오를 늦춰 맞춘다
- **주의(실측 함정)**: 동기화 오프셋은 **녹화/송출 믹스에만** 적용되고 **모니터링에는 적용
  안 됨** — 헤드폰 모니터링으로 맞추면 아무리 값을 바꿔도 "효과 없음"으로 보인다. 반드시
  10초 녹화 후 파일 재생으로 확인
- **NDI > RTSP**: 비디오 소스를 RTSP 미디어 소스 대신 **NDI 소스**(`reframe-out1..4`)로 받으면
  지연이 훨씬 작고 **일정**해서 한 번 맞춘 오프셋이 계속 유효. 실측 지연 **~400ms 근방**
  (RTSP는 mediamtx 릴레이+OBS 네트워크 버퍼링으로 더 크고 흔들림). 남은 ~400ms은 대부분
  reframe 자체 처리 지연(~15fps)이라, 더 줄이려면 OBS NDI 소스 지연 모드 최저 + fps 상향
